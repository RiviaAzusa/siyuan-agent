# 思源笔记定时任务调研报告

## 需求场景

用户在聊天面板输入：「每天下午6点帮我总结今日工作」→ 插件解析意图 → 注册定时任务 → 到时间后自动向 Agent 发送预设指令 → 结果呈现在聊天面板。

## 任务分解

### 1. 自然语言时间解析（NLP → Cron/Schedule）
### 2. 定时调度引擎（Timer Engine）
### 3. 任务持久化（跨重启存活）
### 4. Agent 自动触发（无人值守执行）
### 5. 结果可观测性（用户怎么看到结果）

---

## 一、自然语言时间解析

### 可行性：✅ 高

**方案 A：LLM 提取结构化调度参数（推荐）**

Agent 本身就有 LLM 能力，可直接在 system prompt 或 tool 中让 LLM 输出结构化的调度定义：

```json
{
  "type": "recurring",
  "cron": "0 18 * * *",
  "prompt": "请总结我今天创建和修改的笔记内容",
  "description": "每天下午6点总结今日工作"
}
```

- **优点**：零依赖、天然支持中英文、灵活度最高
- **缺点**：LLM 可能输出不合法 cron，需要校验

**方案 B：专用 NLP 时间解析库**

| 库 | 中文支持 | 大小 | 备注 |
|----|---------|------|------|
| `chrono-node` | ❌ 英文为主 | ~200KB | 最成熟，但中文差 |
| `cron-parser` | N/A | ~20KB | 只解析 cron 表达式，不解析自然语言 |
| 自定义正则 | ✅ | 0 | 覆盖"每天/每周/每月 + 时间"模式 |

**推荐**：方案 A（LLM 提取），配合 `cron-parser` 库校验 cron 表达式合法性。

## 二、定时调度引擎

### 可行性：✅ 中高（有限制需处理）

**核心发现：思源没有面向插件的调度 API**

- 思源后端 `kernel/job/cron.go` 使用 Go `time.Ticker` 做内部定时任务，**不暴露给插件**
- 思源后端 `kernel/task/queue.go` 有 `AppendAsyncTaskWithDelay()`，**也不暴露给插件**
- 插件 SDK (`siyuan.d.ts`) 没有任何 timer/cron/schedule 相关 API

**因此必须在前端（浏览器/Electron）层面实现调度。**

### 方案：前端 setInterval 轮询 + 精确触发

```typescript
class TaskScheduler {
  private checkInterval: ReturnType<typeof setInterval>;
  private tasks: ScheduledTask[] = [];

  start() {
    // 每 30 秒检查一次是否有任务需要执行
    this.checkInterval = setInterval(() => this.tick(), 30_000);
  }

  private tick() {
    const now = Date.now();
    for (const task of this.tasks) {
      if (task.nextRun <= now && !task.running) {
        this.execute(task);
        task.nextRun = this.calcNextRun(task); // 根据 cron 算下次
      }
    }
  }

  stop() {
    clearInterval(this.checkInterval);
  }
}
```

### 关键限制及应对

| 限制 | 影响 | 应对措施 |
|------|------|----------|
| **电脑休眠** | setTimeout/setInterval 暂停，醒来后时间跳变 | tick() 用 `Date.now()` 比对 nextRun，补执行错过的任务 |
| **浏览器标签后台节流** | Chrome 将后台 tab 的 timer 限制到 1次/min | Electron 桌面端不受此限；Web 端可用 Web Worker 规避 |
| **插件重载** | clearInterval 丢失 | onload() 时从持久化存储恢复任务列表 |
| **思源重启** | 所有内存状态丢失 | onLayoutReady() 时恢复 + 检查是否错过执行 |
| **setTimeout 最大值** | 2^31-1 ms ≈ 24.8 天 | 用 setInterval 轮询模式，不依赖单个 setTimeout |
| **时区变化** | 用户旅行时时区变化 | 存储时使用本地时间 + 时区偏移 |

### 可靠性评估

- **桌面端（Electron）**：可靠性高。定时器不受后台节流，30s 轮询精度足够
- **Web 端（浏览器访问）**：可靠性中。后台 tab 节流到 1次/min，可接受；关闭页面则完全失效
- **精度**：±30s（轮询间隔），对"每天下午6点"这类需求完全足够

## 三、任务持久化

### 可行性：✅ 高

**思源插件已有完善的持久化 API：**

```typescript
// 存储路径: /data/storage/petal/{pluginName}/{storageName}
await this.saveData("scheduled-tasks", tasks);  // 保存
const tasks = await this.loadData("scheduled-tasks");  // 读取
```

**数据结构设计：**

```typescript
interface ScheduledTask {
  id: string;                    // 唯一标识
  type: "once" | "recurring";   // 一次性 vs 重复
  cron?: string;                // cron 表达式 (recurring)
  triggerAt?: number;           // 触发时间戳 (once)
  nextRun: number;              // 下次执行时间戳
  prompt: string;               // 要发送给 Agent 的指令
  description: string;          // 人类可读描述
  sessionId?: string;           // 关联的聊天 session
  enabled: boolean;             // 启用/禁用
  createdAt: number;
  lastRunAt?: number;
  lastRunResult?: "success" | "error";
}
```

**恢复流程：**

```
onLayoutReady()
  → loadData("scheduled-tasks")
  → 对每个 enabled 的任务：
    → if nextRun < now: 
        → 补执行？或跳过并计算下一次？（策略可配）
    → 注册到 TaskScheduler
```

## 四、Agent 自动触发

### 可行性：✅ 高

**当前 Agent 的调用链（src/core/agent.ts）：**

```
用户输入 → chat-panel.send(text) → agent.invoke(messages) → LLM + tools → 流式输出
```

**定时触发需要模拟这个流程：**

```typescript
// 在 TaskScheduler.execute() 中
async execute(task: ScheduledTask) {
  // 1. 确保聊天面板存在（可能需要先打开/激活）
  const panel = this.plugin.chatPanel;
  
  // 2. 切换到任务关联的 session（或创建新的）
  if (task.sessionId) {
    await panel.switchSession(task.sessionId);
  } else {
    await panel.newSession();
  }
  
  // 3. 注入系统消息标识这是定时任务
  // 4. 调用 send() 发送预设 prompt
  await panel.send(task.prompt, { isScheduled: true });
}
```

### 关键问题

| 问题 | 方案 |
|------|------|
| **面板未打开** | 后台执行 Agent，不依赖 UI 面板。Agent 核心逻辑（agent.invoke）是纯函数，不依赖 DOM |
| **结果写到哪里** | 写入关联 session 的消息列表，用户下次打开面板时可见 |
| **并发执行** | 如果用户正在聊天，定时任务在独立 session 中执行 |
| **执行失败** | 记录错误到 task.lastRunResult，下次 tick 可重试或跳过 |

**核心洞察**：当前 `agent.ts` 的 `runAgent()` 函数接受 messages 数组，可以脱离 UI 独立调用。只需要收集结果并存储即可。

## 五、结果可观测性

### 可行性：✅ 高

**5 种互补的观测方式：**

#### 1. 聊天面板内查看（主要方式）
- 定时任务的执行结果写入专属或关联的 session
- 用户打开聊天面板 → 看到 session 列表中有新消息标记
- 点击进入查看完整的 Agent 对话记录

#### 2. 思源通知气泡
```typescript
import { showMessage } from "siyuan";
showMessage("⏰ 定时任务「每日工作总结」已完成", 5000, "info");
```
- 任务完成后弹出通知
- 适合提醒用户去查看结果

#### 3. 任务管理面板（新 UI）
- 在设置或独立 tab 中显示任务列表
- 状态：✅ 已完成 / ⏳ 等待中 / ❌ 失败 / 🔄 执行中
- 上次执行时间、下次执行时间、执行结果摘要

#### 4. 文档写入（可选）
- 对于"总结今日工作"类任务，Agent 可直接创建文档
- 用户在笔记本中直接看到生成的文档

#### 5. 系统日志
- `console.log` 级别的调度日志供调试

## 九、风险与限制

| 风险 | 严重度 | 缓解措施 |
|------|--------|----------|
| 思源未运行时无法执行 | 高 | 明确告知用户这是"思源在线时"的定时任务；可在启动时补执行 |
| 电脑休眠错过时间 | 中 | 唤醒后 tick() 自动检测并补执行 |
| Web 端关闭标签页 | 中 | 仅桌面端可靠；Web 端需要后端调度（非插件能力范围） |
| LLM API 到点不可用 | 低 | 重试机制 + 错误记录 |
| 多设备同步冲突 | 低 | 任务只在创建设备执行；或通过设备 ID 区分 |

## 十、结论

**整体可行性：✅ 完全可行，且实现成本不高**

- **核心机制**：前端 setInterval 轮询（30s）+ 持久化存储 + Agent 无人值守调用
- **最大限制**：仅在思源运行时有效（桌面 Electron 环境下可靠性高）
- **工作量预估**：核心调度引擎 + 3 个 Tool + 持久化 ≈ 中等工作量
- **推荐路线**：先实现"LLM 提取 cron + 前端调度 + Session 内展示结果"的最小可用版本
